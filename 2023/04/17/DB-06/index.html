DB C6 Relational Database Design

First Normal Form, 1NF

RECALL

Domain is atomic if its elements are considered to be indivisible units.

  • dealing with non-atomic:

    • Composite attributes : a number of attributes
    • Multi-value attribute : multi-fields / separate table / single field
    • Complex data type : a number of attributes

  • Drawbacks of non-atomic strategy:

    • Complicate storage

    • Encourage redundant storage of data

    • Complicated to query

A relational schema RRR is in first normal form,1NF if the domains of all attributes of RRR are atomic.

  • For the relational database, it’s required that all relations are in 1NF.
  • Atomicity is actually a property of how the elements of the domain are used.

Pitfalls in Relational Database Design

Relational database design requires that we find a “good” collection of relation schemas.

A bad design may lead to:

  • Redundant storage: Wastes space / Easy to result in inconsistency.
  • insert / delete / update anomalies : complicated / many null which is difficult to handle.
  • inability to represent certain information.

Two design methods

  • Top-down
  • Button-up: Universal relation (泛关系) ⇒\Rightarrow⇒ decomposition ⇒\Rightarrow⇒ good database schema

Functional Dependencies

Let RRR be a relation schema, α\alphaα and β\betaβ be attributes, i.e α,β⊆R\alpha,\beta \sube Rα,β⊆R

The functional dependency α→β\alpha \rightarrow \betaα→β holds on RRR if and only if for any legal relations r(R)r(R)r(R), whenever any two tuples t1t_1t1​ and t2t_2t2​ of rrr agree on the attributes α\alphaα, they also agree on the attributes β\betaβ, i.e

∀t1,t2∈r,t1[α]=t2[α]→t1[β]=t2[β]\forall t_1,t_2 \in r, t_1[\alpha] = t_2[\alpha] \rightarrow t_1[\beta] = t_2[\beta] ∀t1​,t2​∈r,t1​[α]=t2​[α]→t1​[β]=t2​[β]

We say β\betaβ is functionally dependent on α\alphaα, α\alphaα functionally determines β\betaβ.

  • Functional dependency is a kind of integrity constraints, expressing the relationship of values on specific attributes.

  • Functional dependency can be used to judge schema normalization and to suggest refinements.

Functional dependency vs. key

  • A functional dependency is a generalization of the notion of a key.

  • KKK is a superkey for the relation schema RRR if and only if K→RK \rightarrow RK→R .

  • KKK is a candidate key for RRR if and only if K→R,¬∃α⊆K,α→RK\rightarrow R ,\lnot\exist\alpha \sube K,\alpha \rightarrow RK→R,¬∃α⊆K,α→R.

  • Functional dependencies allow us to express constraints that cannot be expressed using keys.

Usage

  • Test relations to see if they are legal under a given set of functional dependencies FFF.

    If a relation rrr is legal under a set FFF of functional dependencies, we say that rrr satisfies FFF.

  • Specify constraints (FFF) on the set of legal relations — schema.

    We say that FFF holds on RRR if all legal relations rrr on RRR satisfy the set of functional dependencies FFF.

Given a set of functional dependencies FFF, the set of all functional dependencies logically implied by FFF is the closure of FFF, denoted by F+F^+F+ .

Armstrong’s Axioms
  • reflexivity, 自反律: If b⊆ab \sube ab⊆a,then a→ba \rightarrow ba→b
  • augmentation, 增补律: If a→ba \rightarrow ba→b,then ga→gbga \rightarrow gbga→gb
  • transitivity, 传递律: If a→b,b→ga \rightarrow b,b\rightarrow ga→b,b→g,then a→ga \rightarrow ga→g
  • The reflexivity is trivial.
  • Based on augmentation: If a→ba \rightarrow ba→b,then ga→gbga \rightarrow gbga→gb; By reflexivity, gb→bgb \rightarrow bgb→b; By transitivity, ga→bga \rightarrow bga→b.
  • These rules are
    • Sound, 保真的: generate only functional dependencies that actually hold.
    • Complete, 完备的: generate all functional dependencies that hold.
Additional Armstrong’s Axioms
  • union,合并律: If a→b,a→ca \rightarrow b,a\rightarrow ca→b,a→c, then a→bca\rightarrow bca→bc
  • decomposition, 分解律: If a→bga\rightarrow bga→bg, then a→b,a→ga \rightarrow b,a \rightarrow ga→b,a→g.
  • pseudo transitivity, 伪传递律: If a→b,bg→da\rightarrow b,bg\rightarrow da→b,bg→d, then ag→dag \rightarrow dag→d.

Boyce-Codd Normal Form, BCNF

A relation schema RRR is in BCNF, with respect to a set FFF of functional dependencies, if for all functional dependencies in F+F^+F+ of the form a→ba \rightarrow ba→b , where a⊆Ra \sube Ra⊆R and b⊆Rb \sube Rb⊆R , at least one of the following holds:

  • a→ba \rightarrow ba→b is trival
  • R⊆a+R \sube a^+R⊆a+

Any relation schema with two attributes is in BCNF.

Testing

Testing for BCNF is to check if a non-trivial dependency a→ba \rightarrow ba→b causes a violation of BCNF.

  • Compute a+a^+a+
  • Verify that if R⊆a+R \sube a^+R⊆a+

Simplified Test

If none of the dependencies in FFF causes a violation of BCNF, then none of the dependencies in F+F^+F+ will cause a violation of BCNF either.

However, using only FFF to test for BCNF may be incorrect when testing a relation RiR_iRi​ in a decomposition of RRR.

Third Normal Form, 3NF

A relation schema RRR is in third normal form (3NF) if for all a→ba \rightarrow ba→b in F+F^+F+, at least one of the following conditions holds:

  • a→ba \rightarrow ba→b is trivial
  • R⊆a+R \sube a^+R⊆a+.
  • Each attribute AAA in b−ab - ab−a is contained in a candidate key for RRR. (即A∈b−aA \in b - aA∈b−a是主属性, 若a∩b=ϕa\cap b = \phia∩b=ϕ, 则A = b是主属性).
  • each attribute may be in a different candidate key.
  • 国内其他教材关于3NF的定义: 不存在非主属性对码的部分依赖和传递依赖。即, 当bbb为非主属性时, aaa必须是码; 但当bbb为主属性时, 则aaa无限制。
